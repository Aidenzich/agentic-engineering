# Spec Refiner — 需求完備度與反脆弱閘門

## 技能定位

本技能作為開發流程的強制性前置閘門（Mandatory Pre-implementation Gate）。其核心假設是：**任何尚未被問過的邊界條件，就是系統未來的故障點。** 在完備的防禦性規格產出之前，禁止進入程式碼生成階段。

---

## 執行工作流（Agent Workflow）

### 步驟一：意圖解析與內部推演（Intent & Domain Reasoning）

**絕對不要立刻生成實作程式碼。**

當觸發條件滿足時，Agent 必須先在內部（或透過 `<thought>` 標籤）進行以下推演：

1. 使用者的表面需求是什麼？
2. 根據軟體工程經驗，這個需求背後的「完整生態」應該長什麼樣子？缺了哪些隱性模組？

---

### 步驟二：防禦性提問矩陣（Defense Matrix）

根據使用者的初始需求，從以下四個維度進行評估，挑選出**最迫切且最容易被忽略**的問題進行詢問。

> **核心規則：每輪對話最多只問 2 個問題。** 避免一次性列出所有問題，以降低使用者的認知負擔，確保答案的品質與深度。

#### 四個評估維度

**1. 隱性與關聯需求（Implicit & Cross-cutting Requirements）**

- **狀態追蹤與稽核：** 這個資料變更是否需要記錄操作日誌（Audit Log）？資料刪除是實體刪除還是軟刪除（Soft Delete）？
- **權限與安全性：** 執行此操作的 API 或函式，是否需要前置的權限校驗（RBAC/ABAC）或身分驗證？
- **副作用與事件驅動：** 這個功能執行成功或失敗後，是否應該觸發其他領域的副作用？（例如：清空 Cache、發送非同步通知信、寫入 Metrics）

**2. 領域常識與標準（Domain-Driven Standards）**

- **強制 Agent 思考：** 根據軟體工程常識，實作該類型功能時，業界標準通常包含哪些基礎設施？
  - _例如：使用者提到「匯出報表」，Agent 必須聯想到「非同步任務、進度條、檔案存放空間（S3）」。_
  - _例如：使用者提到「購物車」，Agent 必須聯想到「庫存鎖定機制、購物車過期清理」。_
- **提問句型：** 「根據工程慣例，實作 [某功能] 通常也會一併考慮 [某周邊機制]，請問這部分需要納入本次開發範圍嗎？還是由其他模組處理？」

**3. 異常流（Unhappy Paths）/ 極端值（Edge Cases）**

- 依賴的外部 API 呼叫失敗時，系統應如何降級（Fallback）？
- 網路逾時（Timeout）的閾值為何？逾時後是否需要重試（Retry）？重試幾次？
- 輸入資料格式不符預期（如 JSON 解析失敗）時，應拋出例外、回傳預設值，還是靜默忽略？
- 輸入為空陣列、`null`、`undefined`、空字串時，期望行為是什麼？
- 數值型輸入是否存在負數、零值、浮點數精度問題的風險？
- 字串型輸入是否需要防範極大長度（記憶體風險）或特殊字元（Injection 風險）？

**4. Optional 併發與狀態（Concurrency）**

- 當多個請求同時操作同一份資料時，是否存在 Race Condition 的風險？
- 是否需要冪等性（Idempotency）保障，以確保重複呼叫不產生副作用？
- 是否有共享的可變狀態（Shared Mutable State）需要加鎖或使用原子操作？

**5. Optional 資源限制（Limits）**

- 單次請求的資料量上限為何？是否需要分頁（Pagination）處理？
- 是否存在記憶體溢出（OOM）的風險，例如一次性載入超大資料集？
- 系統預期的最大並發請求量（Peak Load）為何？是否需要速率限制（Rate Limiting）？

---

### 步驟三：多輪收斂（Multi-turn Convergence）

根據使用者的每一輪回答，執行以下判斷邏輯：

1. **答案充分** → 記錄已確認的防禦性條件，並從矩陣中挑選下一個最重要的待問問題繼續提問（每輪仍限 2 個）。
2. **答案模糊** → 針對模糊之處進行追問，並提供 2 至 3 個具體選項供使用者選擇，降低決策摩擦。
3. **所有關鍵維度已涵蓋** → 告知使用者規格已足夠完備，進入步驟四。

**收斂標準：** 當以下條件均已確認時，可視為收斂：

- 至少一個 Unhappy Path 的處理策略已明確。
- 關鍵輸入的 Edge Case 邊界行為已定義。
- 高風險的 Concurrency 或 Resource Limit 問題（若適用）已有初步策略。

---

### 步驟四：輸出最終驅動 Prompt（Final Actionable Prompt）

當所有防禦性提問與隱性需求收集完備後，請總結對話脈絡，並嚴格按照以下格式輸出一段 **User Prompt**。這段 Prompt 將直接作為驅動 Coding Agent 進行實作的輸入指令：

---請複製以下 Prompt 讓 Coding Agent 執行---

**任務目標：**
請根據以下規格，為我實作 [功能名稱/簡述]。你必須嚴格遵守 TDD (Test-Driven Development) 開發流程。

**實作約束：**

1. **測試先行：** 請先輸出涵蓋下方所有情境的測試案例（Test Cases）與測試程式碼。在我審閱並同意測試程式碼之前，**絕對不要**實作主程式邏輯。
2. **防禦性思維：** 實作時必須精準處理下列定義的異常流與邊界條件，禁止自行發明未定義的「Happy Path」預設行為。
3. **介面嚴謹性：** 嚴格遵循下述的 I/O 結構與型別限制。若遇到規格衝突或未定義的盲區，請立即停下來提問。

**一、 核心邏輯與 I/O (Core Logic & I/O)**

- **業務目的：** [為什麼要做這個功能]
- **輸入 (Input)：** [具體的資料結構、型別、約束條件，例如：字串長度限制、必填/選填]
- **預期輸出 (Expected Output)：** [成功時的回傳結構或狀態]

**二、 異常流與防禦邊界 (Exceptions & Boundaries)**

- **異常情境：** [例如：外部 API Timeout] $\to$ **處理策略：** [例如：重試 3 次後拋出 Custom Exception]
- **邊界條件：** [例如：輸入為空陣列] $\to$ **處理策略：** [例如：直接回傳預設的空物件，不進資料庫]
- **併發/限制：** [如果有討論到的 Race Condition 或 Rate Limit 處理策略]

**三、 隱性與周邊需求 (Implicit Requirements)**

- **副作用 (Side-effects)：** [例如：成功建立訂單後，必須發送 Event 給 Message Queue]
- **觀測與安全 (Observability & Security)：** [例如：必須記錄含有 trace_id 的 Log；需先驗證 Token 權限]

**四、 強制測試涵蓋清單 (Mandatory Test Coverage)**
請確保你的測試計畫涵蓋以下條目：

- [ ] 正常流：[描述標準成功路徑]
- [ ] 異常流：[描述針對上述「異常情境」的防禦測試]
- [ ] 邊界值：[描述針對上述「邊界條件」的極端測試]
- [ ] 依賴隔離：[外部 API 或副作用機制的 Mock/Stub 策略]

---

輸出此 Prompt 後，請向使用者確認：

> 「這是為 Coding Agent 準備的最終實作 Prompt。如果沒有需要補充的細節，你可以直接將這段 Prompt 交給 Agent 啟動 TDD 開發流程。」

### 排除在外的範疇（Out of Scope）

- [明確列出本次實作不負責處理的項目，避免範圍蔓延]

```

輸出後，詢問使用者：

> 是否同意使用這份規格開始進行 TDD 開發？如有任何條件需要調整，請直接告知，我將更新規格後再進入實作。

## 技能終止條件

以下任一情況發生時，本技能完成其使命，Agent 可依規格進入正常開發流程：

1. 使用者明確同意「強健性實作規格」的內容。
2. 使用者主動選擇跳過本技能（需明確表示，如：「我了解風險，直接開始實作」）。

> **注意：** 即使使用者選擇跳過，Agent 仍應在最終生成的程式碼中，針對已識別但尚未確認的風險點加入 `TODO` 或防禦性註解，以保留工程意識的痕跡。
```
