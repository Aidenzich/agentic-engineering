# Agentic Engineering Design 團隊筆記

本文件旨在規範團隊在使用與開發 Coding Agent 時的架構決策、工作流程與安全邊界，以最大化開發效率並控制大型語言模型（LLM）可能帶來的認知負擔與風險。

## 一、 核心架構與環境設定

為了確保 Agent 能夠精準且低延遲地操作上下文，我們在工具鏈的選擇上偏向輕量化與高客製化。

- **IDE 與 Agent 核心：Antigravity + 客製化 OpenCode**
  - **決策理由：** 採用開源的 OpenCode 讓我們能完全掌控 Context 注入與 Tool Calling 的底層邏輯。

## 二、 檢索與上下文管理策略

Agent 的上下文精準度決定了程式碼的產出品質。針對專案的檢索策略，我們採取多層次混合模式：

- **基礎檢索首選：`rg` (ripgrep)**
  在多數開發場景中，直接讓 Agent 呼叫 `rg` 進行文字與正則檢索，其速度與直覺性遠勝於複雜的語意搜尋，是解決大部分問題的最有效手段。

- **結構化檢索：AST 與依賴 Graph**
  實作基於 AST 的專案向量化與依賴 Graph 建立功能。這將作為進階的上下文補充工具，專門應對 `rg` 無法處理的深層語意分析或重度重構需求。
- **專案向量化混合檢索**
  可以透過對 codebase 進行向量化後，進行混合檢索，以提高檢索準確度。 讓 Agent 在擁有不具備關鍵字的任務時，也能快速找到相關的程式碼。

## 三、 前端重構與開發工作流

針對前端介面的快速迭代與重構，我們採用「視覺到程式碼」的自動化生成管線。

- **MVP 快速建構流程：**

1. **錄製與總結：** 錄製目標功能的畫面影片，交由 Gemini 進行功能總結與分析。
2. **提示詞工程前置化：** 在進行生成前，透過 System Prompt 嚴格定義 Design System 規範與命名約定，確保生成的產物符合團隊標準。
3. **AI Studio 生成：** 將總結與規範放入 Google AI-Studio Builder 創建 React 專案 MVP。

- **自動化測試掛載：**
- MVP 生成後，首要任務是確保所有 Component ID 的完整對齊與補齊。
- 這些標準化的 ID 將直接對接 Cypress，用於後續 E2E (End-to-End) 自動化測試腳本的撰寫與驗證。

## 四、 認知負擔控制與品質保證

LLM 經常出現「完成單一檔案修改便自信宣稱任務結束」的幻覺，忽略了跨檔案的依賴關係，這會大幅增加人類開發者審查程式碼的認知負擔。

- **開發前提：TDD 驗證驅動**
- 所有開發任務必須以 TDD (Test-Driven Development) 為前提。由 AI 優先撰寫具體的測試案例與程式碼，以此框架嚴格限縮 LLM 後續的功能實作邊界。

- **強制性的架構決策紀錄 (ADR) 檢查表**
- **流程閘門：** 在 Agent 實際動手修改或生成大量程式碼**之前**，必須強制其先產出一份結構化的影響範圍評估（類似簡化的 ADR）。
- **檢查要點：** 該清單必須明確列出「即將修改的檔案」以及「牽連的跨檔案 / 模組依賴關係」。只有在人類確認此檢查表涵蓋了正確的全局視野後，Agent 才能進入編碼階段，從根本上杜絕「單檔幻覺」。

## 五、 基礎設施即程式碼 (IaC) 與安全防護

Agent 生成與執行 IaC (如 Terraform) 具有高風險性，必須在基礎設施層級設置嚴格的護欄。

- **人機協作閘門 (Human-in-the-Loop, HITL)：**
  - Agent 對 Terraform 的自動化操作權限僅止於 `terraform plan`。
  - 任何涉及 `deploy` 或 `destroy` 的狀態變更指令，必須觸發 HITL 機制，交由人類工程師檢視 Plan 報告並手動授權後方可執行。

- **爆炸半徑限制 (Blast Radius Limit)：**
  - **權限閹割：** 透過 IAM 權限邊界或 SCP (Service Control Policies)，嚴格限制 Agent 提權或修改核心網路配置的能力。
  - **資源保護：** 針對 Fargate 或其他自動擴展的運算資源，必須在部署腳本中強制設定 Timeout 與資源用量上限，避免因程序邏輯錯誤陷入無限迴圈，導致資源不當堆積與成本暴增。
- 當進行部署後，必須要人工進行一段時間的成本驗證與功能驗證，確保沒有產生額外的成本與功能異常。

## 六、 可觀測的 Workflow

### 一、 核心量化指標 (Core Telemetry Metrics)

以下兩個判斷依據可以轉化為以下具體的工程遙測 (Telemetry) 指標，並直接埋點在客製化的 OpenCode 或 IDE 擴充套件中：

### 1. 互動摩擦率 (Friction Rate / Turns-to-Success)

- **實作定義：** 計算從「使用者發起初始需求」到「使用者最終接受 (Accept/Commit) 程式碼」之間，經歷了多少次 Prompt 修正對話（Turns）。
- **指標意義：** 這是最重要的品質指標。如果一個任務平均需要 7 到 8 次的來回修正，代表 Agent 的 Zero-shot 能力極差，或者上下文檢索 (如 `rg` / AST) 抓錯了方向。這種情況下，Agent 帶來的「認知負擔與溝通成本」已經大於「自動化省下的時間」，對團隊就是**負提升**。理想的狀態應收斂在 2 到 3 次修正內。

### 2. 有效採用率與依賴度 (Effective Adoption Rate)

- **實作定義：** 追蹤每位開發者每週發起 Agent Session 的次數，以及更關鍵的 **「程式碼保留率 (Code Retention Rate)」**（Agent 生成的程式碼，最終有多少比例存活到 PR 階段而沒被人類刪除或大幅重寫）。
- **指標意義：** 詢問次數代表「嘗試意願」，保留率代表「實際價值」。如果詢問次數高但保留率低，說明大家都在試用，但產出都是垃圾；如果兩者皆高，就證明了正向的 ROI。

### 二、 輔助驗證指標 (Supplementary Metrics)

- **3. 任務放棄率 (Session Abandonment Rate)**
- **實作定義：** 使用者啟動了 Agent 進行某項修改，經過幾輪對話後，最終沒有採用任何 Agent 的產出，直接關閉 Session 並自己手寫。
- **指標意義：** 這通常發生在 Agent 陷入死胡同、反覆鬼打牆（幻覺），或是無法理解跨檔案依賴時。高放棄率是系統架構（尤其是 Context 注入機制）出現嚴重缺陷的明確警訊。

### 三、 如何在工作流中落實？

要收集這些數據，我們不需要額外增加開發者的負擔，可以透過客製化 OpenCode 的機制在背景默默收集：

1. **事件埋點 (Event Tracking)：** 攔截 IDE 中的 `Accept Code`、`Reject`、`Undo` 與 `Commit` 行為，將這些事件與當前的 Agent Conversation ID 綁定。
2. **定期檢討 (Feedback Loop)：** 每週自動拉出「摩擦率最高 (修正次數 > 5)」的 Session Log，團隊在 Code Review 或週會時挑出幾個案例進行覆盤，看看是 System Prompt 沒寫好，還是 `rg` 的檢索策略漏了關鍵檔案。
